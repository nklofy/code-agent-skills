# 代码审查快速检查清单

代码审查时可直接对照本清单快速检查。

## 审查前（2 分钟）

- [ ] 阅读 PR 描述与关联 Issue
- [ ] 检查 PR 规模（理想情况 < 400 行）
- [ ] 确认 CI/CD 状态（测试是否通过）
- [ ] 理解业务目标与变更目的

## 架构与设计（5 分钟）

- [ ] 方案是否匹配问题本身
- [ ] 是否与现有架构模式保持一致
- [ ] 是否存在更简单可行的实现
- [ ] 是否具备扩展性
- [ ] 变更位置是否合理

## 逻辑与正确性（10 分钟）

- [ ] 边界条件是否覆盖
- [ ] 是否有 null/undefined 判定
- [ ] 是否检查了 off-by-one 问题
- [ ] 是否考虑并发/竞态条件
- [ ] 错误处理是否完整
- [ ] 数据类型使用是否正确

## 安全性（5 分钟）

- [ ] 无硬编码密钥/凭证
- [ ] 输入是否校验与清洗
- [ ] 是否防止 SQL 注入
- [ ] 是否防止 XSS
- [ ] 是否有权限校验
- [ ] 敏感数据是否受保护

## 性能（3 分钟）

- [ ] 是否避免 N+1 查询
- [ ] 高开销操作是否优化
- [ ] 大列表是否分页
- [ ] 是否存在内存泄漏风险
- [ ] 是否在合适位置使用缓存

## 测试（5 分钟）

- [ ] 新增/修改代码是否有测试
- [ ] 边界场景是否覆盖
- [ ] 错误路径是否覆盖
- [ ] 测试可读性是否良好
- [ ] 测试是否稳定（可重复）

## 代码质量（3 分钟）

- [ ] 变量/函数命名是否清晰
- [ ] 是否存在重复代码
- [ ] 函数职责是否单一
- [ ] 复杂逻辑是否有必要说明
- [ ] 是否存在魔法数字/字符串

## 文档（2 分钟）

- [ ] 对外 API 是否有文档
- [ ] 如有必要是否更新 README
- [ ] 是否标注破坏性变更
- [ ] 复杂逻辑是否解释清楚

---

## 严重级别标签

| 标签 | 含义 | 处理动作 |
|------|------|----------|
| 🔴 `[blocking]` | 必须修复 | 阻断合并 |
| 🟡 `[important]` | 应该修复 | 若有分歧需讨论 |
| 🟢 `[nit]` | 可优化项 | 不阻断 |
| 💡 `[suggestion]` | 可选替代方案 | 建议评估 |
| ❓ `[question]` | 需澄清 | 请求回复 |
| 🎉 `[praise]` | 正向反馈 | 明确肯定 |

---

## 决策矩阵

| 场景 | 建议结论 |
|------|----------|
| 严重安全问题 | 🔴 阻断并立即修复 |
| 破坏兼容且无迁移方案 | 🔴 阻断 |
| 缺失关键错误处理 | 🟡 建议本轮修复 |
| 新增代码缺少测试 | 🟡 建议本轮补齐 |
| 纯风格偏好 | 🟢 不阻断 |
| 轻微命名优化 | 🟢 不阻断 |
| 能工作但复杂度偏高 | 💡 建议简化 |

---

## 审查时间预算

| PR 规模 | 建议时长 |
|---------|----------|
| < 100 行 | 10-15 分钟 |
| 100-400 行 | 20-40 分钟 |
| > 400 行 | 建议拆分后再审 |

---

## 高风险信号（Red Flags）

出现以下模式时需重点关注：

- 生产代码中保留 `// TODO`
- 遗留 `console.log`
- 被注释掉的历史代码
- TypeScript 中滥用 `any`
- 空 `catch` 块吞错
- Rust 生产代码中滥用 `unwrap()`
- 魔法数字/魔法字符串
- 大段复制粘贴逻辑
- 缺失空值判定
- 硬编码 URL/凭证
