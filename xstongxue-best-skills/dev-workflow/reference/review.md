# 代码审查

开发流程第四步。对实现的代码进行质量检查，发现潜在问题并提出改进建议，为后续 Bug 修复提供输入。

## 使用时机

- 用户提到「代码审查」「Code Review」「检查代码」「看看有没有问题」
- 用户已完成代码实现，准备进行质量检查
- 作为开发流程的第四步被调用（代码实现之后，Bug 修复之前）
- 用户说「修完了」「再看一遍」「帮我复审」等，需要进行回归复审（仅聚焦本轮变更）
- 涉及高风险改动（安全、并发、数据库、公共 API）时，即使用户未明确说“审查”，也应建议进入审查流程

## 不使用时机

- 纯问答/概念解释，无实际代码变更
- 纯文案、排版、命名讨论，且不涉及可执行代码
- 用户明确只要“快速建议”且不需要正式审查报告

## 流程串联

```
需求理解 → 方案设计 → 代码实现 → [代码审查] → Bug 修复
                                    ↑ 当前
```

- 上游输入：来自代码实现的实现代码
- 下游输出：审查报告与改进建议，严重问题交给 Bug 修复处理

## 工作流程

### Step 1：判定审查目标

先明确本次审查对象：

- **远程 PR 审查**：用户提供 PR 编号或 URL
- **本地改动审查**：用户未指定 PR，或明确要求审查当前本地改动

### Step 2：取证与上下文收集

根据目标收集最小必要上下文，避免“只看代码片段就下结论”。

**远程 PR**
- 获取 PR 描述、关联讨论、历史评论
- 明确对比范围（base...head）
- 必要时 checkout 到 PR 分支后再审查

**本地改动**
- 获取变更状态（已暂存 / 未暂存）
- 阅读 diff，确认受影响模块与调用链
- 明确审查边界（本次只审查哪些文件/目录）

### Step 3：预检门禁（Preflight）

优先执行项目已有自动化检查（如可用）：

- lint / typecheck / test / build / preflight

规则：

- 若预检失败，先记录为阻断项，再继续做静态审查
- 若项目无自动化检查，需在报告中明确标注风险

### Step 4：多维度审查

从以下维度进行审查：

**正确性**
- 逻辑是否满足需求与边界条件
- 是否存在空值、越界、状态不一致问题
- 并发/异步场景是否存在竞态（如适用）

**安全性**
- 输入校验与输出编码是否充分（注入、XSS 等）
- 敏感信息是否泄露（密钥、Token、隐私数据）
- 权限与鉴权边界是否清晰

**性能与资源**
- 是否引入明显性能瓶颈
- 是否存在重复计算、无界循环、N+1 查询
- 连接、文件、协程/线程是否正确释放

**可维护性与可读性**
- 命名、模块边界、职责划分是否清晰
- 是否遵循项目规范与目录约定
- 复杂逻辑是否具备必要说明与可测试性

**测试充分性**
- 新增/修改行为是否有对应测试
- 是否覆盖关键分支、错误分支与回归场景

### Step 5：问题分级与阻断规则

将发现的问题按严重程度分类：

| 级别 | 标识 | 含义 | 处理要求 |
|------|------|------|----------|
| 严重 | 🔴 | 功能错误、安全漏洞、破坏兼容性、预检失败 | 必须修复；结论必须为 `Request Changes` |
| 建议 | 🟡 | 存在可维护性/稳健性风险 | 建议本轮修复或说明延期理由 |
| 优化 | 🟢 | 非阻断优化项 | 可后续迭代 |

硬规则：

- 出现任意 🔴，不得给 `Approved`
- 预检失败视为 🔴 阻断项

### Step 6：输出审查报告

按以下模板输出审查报告：

```
## 代码审查报告

### 审查目标
- 类型：本地改动 / 远程 PR
- 范围：xxx
- 审查时间：xxxx-xx-xx xx:xx

### 审查输入证据
- Diff 范围：xxx
- 自动化检查：通过 / 失败（命令与摘要）
- 其他上下文：PR 描述 / 历史讨论 / 设计约束

### 问题清单

#### 🔴 严重问题
**问题1：[问题标题]**
- 位置：`文件路径:行号`
- 问题：[问题描述]
- 风险：[影响范围]
- 建议：[修复建议]

#### 🟡 建议修改
**问题2：[问题标题]**
- 位置：`文件路径:行号`
- 问题：[问题描述]
- 建议：[修改建议]

#### 🟢 可选优化
**问题3：[问题标题]**
- 位置：`文件路径:行号`
- 建议：[优化建议]

### 亮点
- [做得好的点]

### 结论
- 结论：`Approved` / `Request Changes`
- 严重问题：x 个
- 建议修改：x 个
- 可选优化：x 个
- 下一步建议：[明确行动]
```

### Step 7：自动落盘

将本次最终审查报告自动追加写入文档：

- 单模块/未指定模块：写入 `docs/代码审查.md`
- 多模块且已识别模块名：写入 `docs/<module>/代码审查.md`
- 默认目录为 `docs/`，若目录不存在则先创建
- 多模块优先使用 `docs/<module>/` 分目录；若无法确定模块则回退到 `docs/` 根目录
- `<module>` 使用稳定标识（建议用目录名/包名，避免同义词）
- 文件不存在则创建，存在则追加
- 每次新增内容前写入时间标题 `## YYYY-MM-DD HH:mm`
- 粘贴完整报告正文
- 末尾写入 `---`
- 若用户明确指定其他路径，优先按用户路径写入

### Step 8：复审（如需要）

如果用户修改后希望再次审查：

- 仅聚焦本轮修改差异
- 逐项确认历史问题是否关闭
- 检查修复是否引入新问题

## 常见问题检查清单（快速首筛）

**语言通用高频风险**
- [ ] 异常/错误是否被显式处理（避免误吞错误）
- [ ] 资源生命周期是否闭合（文件/连接/锁/监听器/定时器）
- [ ] 并发是否可安全退出（线程/goroutine/任务取消）
- [ ] 类型与边界是否清晰（输入校验、空值/可选值、越界）
- [ ] 是否存在明显性能问题（不必要拷贝、N+1、阻塞关键路径）

**数据库相关**
- [ ] SQL 是否使用参数化查询
- [ ] 是否有 N+1 查询问题
- [ ] 索引与事务边界是否合理

**API 接口**
- [ ] 输入参数是否校验
- [ ] 错误响应是否规范且可观测
- [ ] 是否有必要日志与追踪字段

详细语言规则请按需查看“语言专项审查入口”中的对应文档。

## 注意事项
- 审查反馈要建设性：指出问题时给出可执行建议
- 区分“必须修改”与“偏好建议”，避免强推个人风格
- 对高质量实现要明确给出正向反馈
- 代码审查目标是降低风险、提升可维护性，而非挑刺
- 任何结论都应基于证据（diff、测试结果、复现路径）
- 文档默认中文主叙述；框架、库、API、关键语法名词可保留英文（避免歧义）。
